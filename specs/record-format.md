# Record file format

A record is a markdown file with YAML frontmatter, describing a single
user-facing change. Contributors file one (or more) of these alongside their
PR. At release time, `dv` consumes them into version bumps and CHANGELOG
entries.

## Location

```
.changelog/
  config.yaml
  renames.yaml        # optional — package rename ledger (see below)
  records/
    quiet-cats-sneeze.md
    purple-rhinos-dance.md
    ...
```

The `.changelog/records/` directory holds pending records. When
`dv version` runs, all files there are consumed and the directory is
emptied.

## Filename

Random three-word slugs (e.g., `quiet-cats-sneeze.md`). Generated by
`dv add`. The exact wordlist is an implementation detail; collision risk
should be negligible.

Hand-edited filenames are tolerated but not encouraged — `dv` ignores the
filename's contents and reads everything from the file body.

## Frontmatter

YAML, between `---` delimiters at the top of the file.

### Required fields

| Field      | Type            | Description                                                            |
| ---------- | --------------- | ---------------------------------------------------------------------- |
| `type`     | string          | Conventional Commits type. One of: `feat`, `fix`, `feat!`, `fix!`.     |
| `packages` | list of strings | Package names this record affects. Must match config-known packages.|

A file with `type: feat` and three packages bumps all three by minor.

If a PR genuinely needs different bump levels for different packages (e.g.,
a feature in one, a fix in another), file **two records**. Not one file
with per-package types.

### Optional fields

| Field   | Type            | Description                                                  |
| ------- | --------------- | ------------------------------------------------------------ |
| `links` | list of strings | Issue / PR URLs. Rendered in CHANGELOG if present.           |
| `notes` | string          | Internal notes. Never rendered to CHANGELOG. Reviewer-only.  |

## Body

Free-form markdown. Records are markdown files that should also be valid
standalone documents — recommended convention is to lead with an `# Headline`
line so the file lints cleanly (e.g. markdownlint's MD041 "first line must be
a top-level heading" rule). The CHANGELOG renderer strips the leading `#` and
uses the rest of that line as the bullet.

Bodies without a leading `#` heading still work: the renderer falls back to
the first non-empty line. The h1 convention is a recommendation, not a
hard requirement.

Multi-paragraph bodies are fine; lists, code blocks, links are fine.
**Today only the headline reaches the CHANGELOG bullet** — paragraphs below
the h1 live in the record file for PR reviewers and survive in git history
but are not rendered into CHANGELOG.md. (A future change may surface them
in a long-form companion document.)

## Examples

### A minor feature

```markdown
---
type: feat
packages: [core]
links:
  - https://github.com/example/seshat/issues/42
---

# Add OAuth 2.0 device flow support

Clients without a browser (CLIs, embedded devices) can now authenticate
using the device authorization grant.
```

### A patch fix touching multiple packages

```markdown
---
type: fix
packages: [core, cli]
---

# Correctly handle UTF-8 BOM in config files
```

### A breaking change

```markdown
---
type: feat!
packages: [core]
---

# Drop support for Node 16; minimum version is now Node 20

The `legacy-stream` API has been removed; use `WebStream` instead.
```

## Package renames (`renames.yaml`)

When a package is renamed, records authored under its old name still
reference it. Rather than guess at renames heuristically (see `design.md`
§ Package renames and deletions for why that's avoided), `dv` resolves
references through an explicit, append-only ledger at
`.changelog/renames.yaml`:

```yaml
- from: core
  to: engine
  at: "1.3.0"      # engine's first version under the new name
```

A record that says `packages: [core]` is resolved to `engine` via this
ledger. The record file itself is never rewritten — it stays a faithful
record of what was authored; the ledger does the mapping at processing
time.

Entries can be added by hand or via `dv rename <old> <new>`, which is pure
sugar: it appends to the ledger and does nothing else. Renaming the actual
package (moving the directory, changing the manifest `name`) is done with
your ecosystem's normal tools — `dv` only records the lineage.

A reference to a package that discovery can't find *and* has no rename
entry is an **unresolved reference**: `dv version` halts with guidance (or drops the
record with `--prune` if the package was genuinely deleted).

## Validation

`dv validate` checks:

- Frontmatter parses as YAML
- `type` is one of the allowed values
- Every package in `packages` is known to the config (after applying
  `renames.yaml`)
- File is in `.changelog/records/` (not nested deeper)
- Body is non-empty

Validation runs in CI and as a pre-commit hook (opt-in).
