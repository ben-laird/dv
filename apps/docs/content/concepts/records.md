# Records

A **Record** is a small Markdown file that declares one change. It's the
unit dv reads to compute what to release; everything downstream — the
SemVer bump, the CHANGELOG entry, the commit message — flows from the
Records that have accumulated since the last release.

This page explains what a Record is, why dv reads them instead of commit
messages, and how they interact with the rest of the system. For the
exact file format, see the [Records reference](/reference/record-format).

## What a Record looks like

```markdown
---
type: feat
packages:
  - "@my/api"
notes: add /v2 endpoint with pagination
---

## Add /v2 endpoint with pagination

The new endpoint accepts a `cursor` query param and returns paginated
results in the `Link` header. Old `/v1` clients keep working — the
endpoint is additive.
```

Three parts:

1. **Frontmatter** — a YAML block declaring the change type and the
   packages affected.
2. **Notes** (in frontmatter) — a one-line summary that ends up in the
   CHANGELOG bullet.
3. **Body** (optional) — long-form Markdown that supplies the prose for
   `HISTORY.md` (the long-form companion to CHANGELOG, if you enable
   it).

Records live in `.dv/records/`. Filenames are generated by `dv add` from
a wordlist (`sunset-cliff-roam.md`) so collisions across branches are
vanishingly unlikely. You can also write them by hand — the format is
all that matters.

## The Record-type vocabulary

A Record's `type` field is drawn from a small, fixed set:

| Type | Meaning | Stable bump | Unstable (pre-1.0) bump |
|---|---|---|---|
| `feat` | New feature | minor | minor |
| `fix` | Bug fix | patch | patch |
| `feat!` | Breaking new feature | major | minor (capped) |
| `fix!` | Breaking bug fix | major | minor (capped) |

That's the entire vocabulary. Things you might expect but won't find:

- **No `chore`, `docs`, `refactor`, `style`, `test`, `perf`, etc.**
  These are conventional-commit categories that don't produce a version
  bump. Internal churn belongs in git history, not in the CHANGELOG.
- **No custom types.** The vocabulary is locked. If you find yourself
  wanting `chore!`, what you actually want is `fix!` or `feat!` —
  the change has user-visible impact, which is the whole criterion.

This is deliberate. The fewer types there are, the fewer decisions
contributors have to make, and the more uniform the CHANGELOG ends up.

## Why Records, not commit messages

dv could read your commit log. It deliberately doesn't. Three reasons:

### Commit messages aren't stable

A commit message goes through a lot before it lands:

- The author writes it.
- A commit-msg hook may rewrite it.
- A PR template may inject sections.
- A squash merge collapses everything into one message that no
  contributor authored.
- The reviewer may rewrite it during merge.

By the time a tool reads `git log`, the message bears only a loose
relationship to the author's original intent. **Records survive all of
that** — they're files in the tree, diff-able and reviewable like any
other change.

### Style enforcement is the wrong abstraction

Conventional Commits parsing makes the *commit message format* a
load-bearing artifact. That means every PR triggers a style fight:

- A commit hook rejects `add stuff`.
- CI rejects `Fix bug` (wrong tense).
- A linter rewrites `chore: foo` to `chore(scope): foo` because someone
  added a scope requirement six months ago.

This is real friction with no payoff. The actual question — "what
changed?" — is something the *author* knows; they should answer it
directly, not encode it in a message that a parser will misread.

### Records are co-located with the change

A Record sits in `.dv/records/` next to the code change it describes.
Reviewers see it in the PR diff. It can reference the body code (the
file path, the function name). It's editable in subsequent commits —
fix a typo, add detail, no force-push needed. Try doing any of that
with a commit message.

## Records and packages

A Record can affect more than one package:

```markdown
---
type: feat!
packages:
  - "@my/api"
  - "@my/client"
notes: rename `/users` to `/people` (breaking)
---
```

This is one *change* with effects in two packages. Both get the same
bump (`feat!`), both get a CHANGELOG entry, both consume this Record.
You're not duplicating work; you're declaring a single coherent change
once.

The `packages` field references packages by name (the same name that
appears in your manifests). Names dv has never heard of are
**Unresolved References** — dv halts with a clear error unless you
explicitly opt into pruning them (`--prune`) or recording a rename
(`dv rename <old> <new>`).

## How dv reads Records

When you run `dv version` (or its preview, `dv status`), dv:

1. **Lists Records.** Every `.md` file in `.dv/records/` gets parsed.
2. **Resolves package references.** Each package in each Record is
   checked against the discovered package set, with the rename ledger
   applied. Unknown packages become Unresolved References.
3. **Classifies each Record's type** into a SemVer bump, accounting for
   the package's current stability (pre-1.0 vs. ≥1.0).
4. **Aggregates per package.** Multiple Records targeting the same
   package join into one bump — the max of the individual bumps.
   (Three `fix`es plus one `feat` against the same package → minor.)
5. **Builds the Plan.** A side-effect-free description of every bump,
   every CHANGELOG entry, every dependency constraint to rewrite.

The Plan is what `dv status` and `dv version --dry-run` print. It's
also what `dv version` *executes* — same code path, same Plan, no
chance of drift between preview and reality. See [Two-phase release](/concepts/two-phase-release)
for the execution model.

## Records have a lifecycle

A Record exists from the moment it's authored until `dv version`
consumes it:

```
                ┌─────────┐
                │ author  │  ← dv add  or  manual write
                │ Record  │
                └────┬────┘
                     │  git commit (Record + code together)
                     ▼
                ┌─────────┐
                │ pending │  ← dv status / dv version --dry-run
                │ Record  │     can preview the effect
                └────┬────┘
                     │  dv version
                     ▼
                ┌──────────┐
                │ consumed │  ← Record is *deleted* by dv version,
                │ (deleted)│     its content lives on in CHANGELOG.md
                └──────────┘
```

A few non-obvious consequences:

- **Records aren't a permanent history.** They're a transient buffer
  between "change happened" and "release shipped". The permanent
  history is the CHANGELOG.
- **Idempotence is automatic.** Running `dv version` with no pending
  Records is a no-op. Running it twice in a row does nothing the second
  time.
- **Merge conflicts are rare and harmless.** Two branches each
  authoring a Record collide only if they pick the same filename —
  vanishingly rare given the wordlist generator. When it does happen,
  rename one of them; they're independent.

## When to author a Record

A simple rule: **author a Record for any change that should appear in
the CHANGELOG**.

- New feature → `feat` Record
- Bug fix that users will notice → `fix` Record
- Breaking change → `feat!` or `fix!` Record (whichever shape fits)
- Internal refactor with no user-visible effect → no Record
- Doc update → no Record
- Test additions → no Record

The criterion is *user-visible impact*. If a downstream consumer would
care, it deserves a Record. If they wouldn't, it doesn't.

When in doubt: a Record costs you 30 seconds of `dv add`. Not having
one costs the user reading your CHANGELOG.

## Common patterns

**One change, multiple packages** — one Record with all affected
packages listed.

**Renames** — use `dv rename <old> <new>` to record the lineage edge.
Records referencing the old name then resolve cleanly. (See the
[CLI reference](/reference/cli) for `dv rename`.)

**Records authored against a package that gets renamed mid-flight** —
the rename ledger handles this. You don't have to edit your old Records.

**Many small Records vs. one big one** — small is better. Each Record
becomes one CHANGELOG bullet. Three `fix` Records read better in the
output than one Record listing three things.
